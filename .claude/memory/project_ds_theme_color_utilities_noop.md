**HARD FACT (verified against built CSS bundle by Ingrid, P4W4 2026-06-12 — not reasoning):** the noorinalabs-isnad-graph frontend Tailwind build generates **ZERO atomic color/background utility rules**. `bg-card`, `bg-background`, `bg-primary`, `border-border`, `text-foreground`, `text-muted-foreground`, `bg-accent`, `bg-muted` all produce **0 rules in dist** → they are no-ops in the shipped app. Example latent bug: `AuthCallbackPage`'s error card `className="bg-card border-border"` renders with NO background/border.

**Root cause (CORRECTED — earlier ":root" assumption was wrong; verified against the installed tarball `node_modules/@noorinalabs/design-system@0.0.5-wave4.0/dist/styles.css`):** the consumed DS dist defines semantic `--color-*` in `@theme{ }` (a Tailwind BUILD at-rule — runtime-ignored by browsers) **plus** `[data-theme=light]{ }` / `[data-theme=dark]{ }` plain-CSS blocks (runtime-valid, attribute-gated) — but **NOT** unconditional `:root`. The DS dist is shipped pre-compiled, so the app's Tailwind never reads the DS `@theme{}` → never learns `card`/`foreground`/`border` as color names → emits no color utilities. (DS#104/PR#107 intended to move these to `:root` but that **never shipped** — no version bump + publish-guard skips existing version; DS main=0.0.2 vs consumed 0.0.5-wave4.0. Tracked: **DS#111**.)

**Runtime resolution of `var(--color-*)`:** works via the `[data-theme]` blocks because `src/hooks/useTheme.ts` ALWAYS sets `data-theme` on `<html>` on mount (system→light/dark). So `style={{color:'var(--color-success)'}}` AND the bridged `.bg-card{background:var(--color-*)}` both resolve at runtime — gated on `data-theme` being set (true post-mount; a brief pre-mount FOUC instant is the only gap, pre-existing). NOT dependent on `:root`.

**Trap:** a naive token→`bg-*`/`text-*` Tailwind-utility conversion SILENTLY emits 0 rules (Tailwind didn't learn the colors) — until the bridge below. Distinct from resolution: once a `var(--color-*)`-bearing rule exists (inline OR bridged utility), it resolves via `[data-theme]`. Safe-without-bridge: Tailwind NON-color utilities (spacing/radius/layout — DS scale==Tailwind scale) + DS **component** classes from dist (`.stat-card`/`.badge-*`/`.btn-primary`/`.feature-card`/`.muted-text`/`.error-text`/`.text-success`…), all of which already use `var(--color-*)` and render on live stg today.

**Fix — SHIPPED P4W4 (ig#1000 / PR #1002, merged f3fffb4; owner: do-the-right-thing per [[feedback_prefer_correct_over_expedient]]):** app-side **`@theme inline` bridge** in `frontend/src/styles/theme.css` — registers all 65 DS semantic/domain `--color-*` as Tailwind theme colors referencing the runtime `var(--color-*)` (the `inline` keyword is load-bearing: it emits `var()` into utilities rather than freezing the value, so light/dark adaptivity flows from `[data-theme]`). App-only (no DS change needed for FUNCTION — colors resolve via `[data-theme]`; DS `:root` is a separate robustness fix in DS#111). Activated ~200 previously-inert color-utility usages across ~14 files — reviewer-confirmed all intended corrections (AuthCallback error card, SearchPage dropdown, ProtectedRoute scrim now paint; zero gradient/bg-image surfaces → no layering risk). **Dispositive that the bridge works against the PINNED dep:** existing `.stat-card` etc. use the identical `var(--color-*)` and render today, so the bridged utility resolves identically. Junseo's ig#1002 review flagged "inert/transparent" (saw `@theme`, missed `[data-theme]`) — false alarm on the conclusion, but his adjacent DS-publish catch was real → DS#111. ig#981 then migrates the 3 heaviest inline-style files onto the now-working utilities. **`className="text-success"` now WORKS** — trap closed P4W4.
